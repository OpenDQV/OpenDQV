# Community Use Cases

We are building this list with the community. If you are using OpenDQV, we would love to hear about it.

## Share Your Use Case

Open a [GitHub Discussion](https://github.com/OpenDQV/OpenDQV/discussions) with the label **use-case** and tell us:

- **Industry / domain** — what sector are you in?
- **What you're validating** — customer records, transactions, sensor data, forms, API payloads?
- **The problem you were solving** — what was breaking before, and what changed?
- **Scale** — rough volume (records per day, number of source systems)
- **Anything you had to work around** — gaps in OpenDQV that required a workaround

You do not need to share company names, numbers, or anything confidential. A single paragraph describing the pattern is enough and is genuinely useful for other community members.

## Why This Matters

Real use cases from real deployments help us:

- Prioritise the roadmap based on actual pain points, not assumptions
- Write better example contracts for industries we haven't covered yet
- Understand where the documentation is unclear or incomplete
- Justify the project to contributors who want to know it is being used

## What We Have Heard So Far

OpenDQV launched on 2026-03-21. **You could be the first case study here.**

If you have deployed OpenDQV — even as a proof of concept, even in a single pipeline — share it via a [GitHub Discussion](https://github.com/OpenDQV/OpenDQV/discussions) with the label **use-case**. We will document it here, with your permission, for every practitioner who comes after you wondering whether this tool works for their domain.

Every mature open-source project started with one person willing to say "I tried it and here is what happened." Check the [Discussions tab](https://github.com/OpenDQV/OpenDQV/discussions) for the latest.
